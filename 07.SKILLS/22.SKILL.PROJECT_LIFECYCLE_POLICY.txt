==========
01.00_OBJETIVO
==========

Establecer un flujo operativo único, repetible y auditable para cualquier sesión de trabajo.

Objetivos:

- evitar pérdida de contexto;
- asegurar continuidad humano/IA;
- mantener consistencia entre artefactos;
- evitar burocracia permanente durante la ejecución;
- concentrar actualizaciones de continuidad en el cierre formal;
- garantizar trazabilidad sin inventar estado.

==========
02.00_DEFINICION_DE_EVENTOS
==========

Eventos del ciclo de vida:

- INICIO_SESION
- EJECUCION
- CIERRE_SESION

Cada evento tiene:

- inputs;
- acciones permitidas;
- acciones obligatorias;
- outputs esperados;
- límites.

El cierre formal solo se activa con frase explícita definida en este Skill.

==========
03.00_FLUJO_DEL_CICLO_DE_VIDA
==========

Secuencia base:

INICIO_SESION
→ EJECUCION
→ CIERRE_SESION

Regla corregida v1.1:

Durante INICIO_SESION y EJECUCION no es obligatorio actualizar BATON, backlogs,
deuda técnica ni HUMAN.

Estas actualizaciones se realizan:

- al CIERRE_SESION formal;
- o cuando el humano lo solicita de forma explícita;
- o cuando el cambio doctrinal exige actualizar HUMAN.

Esto evita mantenimiento prematuro, duplicación de esfuerzo y ruido documental durante la ejecución.

==========
04.00_INICIO_DE_SESION
==========

INPUTS RECOMENDADOS:

- BATON principal del proyecto activo, si existe;
- HUMAN vigente, si aplica;
- WhoAmI vigente, si aplica;
- RADAR_INDEX o RADAR equivalente, si aplica;
- backlog operativo del proyecto activo, si existe.

ACCIONES:

- leer BATON si existe;
- leer WhoAmI si existe y aporta continuidad humana;
- entender estado actual;
- identificar NEXT_ACTION;
- validar coherencia con HUMAN si el proyecto tiene capa HUMAN;
- cargar contexto mínimo suficiente.

PROHIBIDO:

- asumir estado sin evidencia;
- modificar archivos solo por iniciar sesión;
- usar RADAR_FULL sin justificación;
- actualizar BATON, backlogs, deuda técnica o HUMAN solo por iniciar sesión.

OUTPUT:

- comprensión clara del estado;
- plan inicial o siguiente acción;
- brechas declaradas si faltan insumos.

==========
05.00_EJECUCION
==========

Durante la ejecución se prioriza avanzar trabajo real.

TIPOS DE CAMBIO:

----------
05.10_CAMBIO_OPERATIVO
----------

Ejemplos:

- avanzar tarea;
- corregir bug;
- crear script;
- ejecutar diagnóstico;
- validar output;
- proponer siguiente minibattle.

Regla:

No obliga por sí mismo a actualizar BATON, backlog, deuda técnica ni HUMAN durante la sesión.

La actualización de continuidad se difiere al cierre formal, salvo instrucción explícita del humano.

----------
05.20_CAMBIO_ESTRUCTURAL
----------

Ejemplos:

- redefinir arquitectura;
- cambiar estructura de carpetas;
- cambiar precedencia de artefactos;
- modificar el modelo de gobierno del proyecto.

Regla:

Si el cambio altera doctrina, intención humana, precedencia o regla raíz, debe evaluarse actualización de HUMAN.
Si solo altera operación, no se actualiza HUMAN por defecto.

----------
05.30_CAMBIO_DE_REGLAS
----------

Ejemplos:

- nuevo Skill;
- modificación de Policy;
- modificación de Control;
- cambio en GRC (Governance, Risk and Compliance);
- cambio en protocolo de cierre.

Regla:

Debe actualizarse el artefacto rector correspondiente.
Si el cambio afecta ciclo de vida, este Skill debe mantenerse alineado.

----------
05.40_REGLAS_GENERALES_DE_EJECUCION
----------

- no mezclar responsabilidades;
- no duplicar información;
- mantener trazabilidad;
- declarar brechas;
- no inventar estado;
- ejecutar validación cuando corresponda;
- mantener el mantenimiento documental para cierre formal, salvo instrucción explícita.

==========
06.00_CIERRE_DE_SESION_POR_ACTIVACION_EXPLICITA
==========

----------
06.10_FRASE_ACTIVADORA
----------

El cierre formal de sesión solo se activa cuando el humano usa la frase exacta:

"cierra sesión"

Otras frases, consultas de estado o solicitudes parciales no activan el cierre formal.

Ejemplos que NO activan cierre formal:

- "en qué estamos";
- "qué falta";
- "resumen";
- "estado del proyecto";
- "actualiza BATON";
- "actualiza backlog";
- "dame deuda técnica".

Estas solicitudes pueden requerir respuesta o actualización parcial, pero no ejecutan cierre formal completo.

----------
06.20_CONTRATO_DE_CIERRE
----------

Al activarse el cierre formal, la IA debe entregar y/o preparar:

1. Deuda técnica.
2. Backlogs.
3. Qué hemos hecho.
4. Qué falta.
5. En qué estamos.
6. Path accionable de desarrollo: minibattles -> battles -> entregable funcional a gerencia.
7. Actualización del BATON principal del proyecto activo.
8. Actualización del backlog operativo del proyecto activo, si existe.
9. Validación operativa.
10. Estado de cierre.
11. Handoff opcional para próxima sesión.

----------
06.30_REGLA_DE_GENERALIDAD
----------

Este Skill es general.

No debe asumir rutas exclusivas de SkillsMachine.
No debe asumir nombres específicos de proyecto.
No debe asumir que siempre existe BATON.
No debe asumir que siempre existe backlog.
No debe asumir que siempre existe HUMAN.
No debe asumir que siempre existe RADAR.

Debe operar con los artefactos equivalentes del proyecto activo.

----------
06.40_BATON_EN_CIERRE
----------

La actualización de BATON debe apuntar al BATON principal del proyecto activo.

Si existe más de un BATON, la IA debe identificar el principal antes de actualizar.
Si no puede identificarlo con evidencia, debe declararlo como brecha y pedir instrucción.

----------
06.50_BACKLOG_EN_CIERRE
----------

La actualización de backlog debe apuntar al backlog operativo del proyecto activo, si existe.

Si existen varios backlogs, la IA debe usar el backlog operativo principal.
Si no puede identificarlo con evidencia, debe declararlo como brecha y pedir instrucción.

----------
06.60_DEUDA_TECNICA_EN_CIERRE
----------

La deuda técnica debe registrarse como backlog operativo, salvo que el proyecto tenga un artefacto formal de deuda técnica.

Debe distinguirse:

- deuda técnica;
- backlog funcional;
- brecha de información;
- riesgo abierto.

----------
06.70_HUMAN_EN_CIERRE
----------

HUMAN solo se actualiza cuando existe:

- cambio doctrinal;
- cambio de precedencia;
- cambio de intención humana;
- cambio de regla raíz;
- instrucción explícita del humano.

HUMAN no se actualiza por cambios operativos normales.

----------
06.80_PATH_MINIBATTLES_BATTLES_ENTREGABLE
----------

El cierre debe incluir path accionable de desarrollo:

MINIBATTLE -> BATTLE -> ENTREGABLE_FUNCIONAL_A_GERENCIA

Formato recomendado:

MINIBATTLE | OBJETIVO | OUTPUT | BATTLE | ENTREGABLE_GERENCIA | ESTADO

Si faltan fechas, responsables o criterios de éxito, no se inventan.
Se declaran como pendientes solo si son necesarios para gestión.

----------
06.90_EVIDENCIA_Y_NO_INVENCION
----------

No se debe inventar estado.

Si falta evidencia, la IA debe declararlo como brecha.

Hechos confirmados, inferencias y pendientes deben distinguirse.

==========
07.00_ACTUALIZACION_DE_ARTEFACTOS
==========

----------
07.10_REGLA_BASE
----------

Las actualizaciones de continuidad se hacen al cierre formal o bajo instrucción explícita.

Artefactos típicos:

- BATON principal del proyecto activo;
- backlog operativo del proyecto activo;
- deuda técnica;
- HUMAN, solo si aplica;
- WhoAmI, solo si cambia contexto humano operacional;
- RADAR, si aplica;
- readiness o equivalente de cierre, si existe.

----------
07.20_NO_ACTUALIZACION_PERMANENTE
----------

Antes y durante la sesión no se obliga a actualizar:

- BATON;
- backlogs;
- deuda técnica;
- HUMAN.

Esto no prohíbe actualizar estos artefactos si el humano lo solicita explícitamente.

----------
07.30_CRITERIO_DE_ACTUALIZACION
----------

Actualizar solo si:

- hubo cambio real;
- hay evidencia suficiente;
- el artefacto existe o el humano aprueba crearlo;
- el cambio no duplica información innecesaria.

==========
08.00_VALIDACION_CON_RADAR
==========

----------
08.10_REGLA_GENERAL
----------

RADAR es mapa/evidencia de filesystem, no fuente de verdad lógica.

RADAR se usa para:

- validar rutas;
- detectar archivos creados, modificados o eliminados;
- revisar estado documental;
- apoyar cierre de sesión;
- preparar continuidad para IA en blanco.

----------
08.20_CUANDO_PEDIR_RADAR
----------

Si falta RADAR vigente o evidencia suficiente para cerrar con fidelidad, la IA debe pedir ejecutar RADAR antes de completar el cierre formal.

Esto aplica especialmente cuando:

- hubo cambios de filesystem;
- se crearon archivos;
- se modificaron artefactos;
- se eliminó o movió contenido;
- el cierre requiere confirmar rutas;
- el proyecto depende de RADAR como mapa de continuidad.

----------
08.30_CUANDO_RADAR_PUEDE_NO_APLICAR
----------

RADAR puede no aplicar si:

- la sesión fue puramente conceptual;
- no hubo cambios de filesystem;
- el proyecto no usa RADAR ni equivalente;
- el humano pide cierre parcial y acepta la brecha.

En ese caso se debe declarar explícitamente que RADAR no fue usado.

==========
09.00_RELACION_CON_GIT
==========

Este Skill no define comandos Git.

Toda operación Git debe derivarse al Skill de Git aplicable del proyecto activo.

Ejemplos de materias derivadas al Skill de Git:

- git status;
- git diff;
- git add;
- git commit;
- git push;
- ramas;
- tags;
- reversión;
- staged vs unstaged;
- archivos ignorados;
- commits atómicos.

Durante cierre, este Skill puede exigir verificar estado de versionamiento,
pero no redefine cómo hacerlo.

==========
10.00_REGLAS_DE_CONSISTENCIA
==========

Caso 1:
BATON contradice HUMAN.

Dictamen:
BATON está incorrecto o desactualizado.

Acción:
Actualizar BATON o declarar brecha.

----------

Caso 2:
Skill contradice HUMAN.

Dictamen:
Conflicto conceptual.

Acción:
Resolver actualizando explícitamente HUMAN o Skill según decisión humana.

----------

Caso 3:
RADAR contradice BATON.

Dictamen:
BATON está desactualizado o RADAR refleja cambios no incorporados.

Acción:
Revisar evidencia antes de cerrar.

----------

Caso 4:
IA propone acción que viola Skill.

Dictamen:
Acción inválida.

Acción:
Bloquear o pedir excepción humana documentada.

----------

Caso 5:
Usuario pide estado pero no cierre.

Dictamen:
No activar cierre formal.

Acción:
Responder estado, pero no ejecutar protocolo completo.

----------

Caso 6:
Usuario dice "cierra sesión".

Dictamen:
Activar cierre formal completo.

Acción:
Ejecutar contrato de cierre de este Skill.

==========
11.00_ANTIPATRONES
==========

- Actualizar BATON en cada microavance sin necesidad.
- Convertir el cierre en burocracia permanente.
- Usar RADAR como fuente de verdad lógica.
- Cerrar sesión sin declarar brechas.
- Inventar estado para completar cierre.
- Actualizar HUMAN por cambios operativos normales.
- Mezclar Git dentro de este Skill.
- Duplicar información entre BATON, backlog y HUMAN.
- Crear backlog si el proyecto ya tiene uno operativo suficiente.
- Omitir path minibattles -> battles -> entregable funcional a gerencia al cierre formal.
- Tratar "estado del proyecto" como "cierra sesión".

==========
12.00_DEFINICIONES_OPERATIVAS
==========

----------
12.10_MINIBATTLE
----------

Unidad corta de avance, verificable, con output concreto y evidencia mínima.

Debe reducir riesgo, desbloquear trabajo o producir avance funcional.

----------
12.20_BATTLE
----------

Bloque mayor de trabajo compuesto por una o más minibattles.

Debe conducir hacia una capacidad funcional, decisión relevante o entregable demostrable.

----------
12.30_ENTREGABLE_FUNCIONAL_A_GERENCIA
----------

Resultado demostrable, entendible por gerencia y con valor visible.

Puede ser demo, reporte, tablero, presentación, proceso operativo validado o automatización funcional.

No debe ser solo documentación interna salvo que el objetivo gerencial sea gobierno documental.

----------
12.40_BATON_PRINCIPAL_DEL_PROYECTO_ACTIVO
----------

Archivo o artefacto de continuidad que resume el estado operativo mínimo para continuar el proyecto en una IA en blanco.

----------
12.50_BACKLOG_OPERATIVO_DEL_PROYECTO
----------

Lista activa de pendientes, mejoras, deuda técnica, riesgos y próximas acciones del proyecto.

Se actualiza al cierre si existe y si hay información suficiente.

----------
12.60_RADAR
----------

Mapa o índice vigente del filesystem/proyecto usado como evidencia de estado, rutas y artefactos disponibles.

Si falta RADAR y el cierre depende de estado de archivos, la IA debe pedir su ejecución.

==========
13.00_ACUMULACION_Y_PERFECCIONAMIENTO_DE_ARTEFACTOS
==========

----------
13.10_REGLA_BASE
----------

Durante el cierre formal de sesión, cada artefacto debe cumplir su función correcta.

No se debe copiar la misma información en todas las capas.

Regla de capa:

- Backlog acumula.
- Deuda técnica acumula.
- HUMAN se perfecciona.
- BATON se refresca.
- RADAR se regenera.
- Git registra.

----------
13.20_BACKLOG
----------

El backlog operativo del proyecto acumula pendientes no resueltos, mejoras, decisiones abiertas y próximos pasos.

En cada cierre formal:

- se agregan nuevos pendientes;
- se actualizan pendientes existentes;
- se cierran ítems resueltos;
- se fusionan duplicados;
- se reclasifican ítems cuando cambia su naturaleza.

Un pendiente no resuelto permanece abierto.

No debe duplicarse como nuevo ítem si ya existe.

----------
13.30_DEUDA_TECNICA
----------

La deuda técnica acumula problemas técnicos, estructurales u operativos no resueltos.

Debe distinguirse de:

- backlog funcional;
- brecha de información;
- riesgo abierto;
- decisión pendiente.

Estados recomendados:

- NUEVA
- EXISTENTE_SIN_CAMBIO
- EXISTENTE_ACTUALIZADA
- MITIGADA
- RESUELTA
- ACEPTADA_CONSCIENTEMENTE
- DESCARTADA

La deuda técnica debe mantener, cuando exista evidencia suficiente:

- descripción;
- causa;
- impacto;
- riesgo;
- evidencia;
- próximo paso;
- prioridad;
- relación con minibattle o battle si aplica.

----------
13.40_HUMAN
----------

HUMAN no acumula tareas ni pendientes operativos.

HUMAN se perfecciona solo cuando existe:

- cambio doctrinal;
- aclaración de intención humana;
- cambio de visión;
- cambio de principio rector;
- cambio de precedencia;
- cambio de regla raíz;
- mejora conceptual canónica.

Ejemplo correcto:
Actualizar HUMAN para declarar una nueva regla superior de arquitectura o gobernanza.

Ejemplo incorrecto:
Agregar a HUMAN una tarea pendiente de implementación.

Las tareas pendientes van en backlog, no en HUMAN.

----------
13.50_BATON
----------

BATON no acumula historia completa.

BATON es snapshot operativo vigente.

Debe reflejar:

- estado actual;
- objetivo activo;
- próximos pasos;
- riesgos o brechas que afectan continuidad;
- archivos/capas relevantes para continuar;
- decisión operativa vigente.

BATON debe refrescarse al cierre formal cuando exista información suficiente.

No debe convertirse en manual, backlog ni changelog completo.

----------
13.60_RADAR
----------

RADAR no se edita manualmente.

RADAR se regenera como evidencia del filesystem o estructura equivalente del proyecto.

RADAR no decide lógica ni doctrina.

RADAR ayuda a validar:

- archivos existentes;
- rutas;
- cambios de filesystem;
- outputs;
- estado documental observable.

----------
13.70_GIT
----------

Git registra trazabilidad histórica mediante commits, ramas, tags y diferencias.

Este Skill no define comandos Git.

La operación Git debe derivarse al Skill de Git aplicable.

----------
13.80_REGLA_ANTI_DUPLICACION
----------

No se debe duplicar la misma información en múltiples artefactos.

Cada contenido debe vivir en su capa correcta:

- doctrina en HUMAN;
- estado operativo vigente en BATON;
- pendientes en backlog;
- problemas estructurales en deuda técnica;
- evidencia filesystem en RADAR;
- trazabilidad histórica en Git.

Si una misma idea aparece en varias capas, debe tener propósito distinto en cada una.

----------
13.90_CRITERIO_DE_CIERRE
----------

Al cierre formal, la IA debe revisar si cada dato pertenece a:

- HUMAN;
- BATON;
- backlog;
- deuda técnica;
- RADAR;
- Git;
- otro artefacto del proyecto activo.

Si no existe evidencia suficiente para clasificar un dato, debe declararse como brecha y no inventarse.

MB-GRC-026F1_CONTINUITY_REFRESH_RULE
==========

FECHA.............: 2026-06-04 10:35:38
ESTADO............: ACTIVE_RULE

REGLA:
Cuando una minibattle, cierre de sesión o cambio operacional afecte continuidad,
BATON y WHOAMI deben actualizarse juntos antes del commit final.

NO_RECOMENDADO:
- Commit BATON-only.
- Commit WHOAMI-only.
- Refrescar uno y luego corregir el otro en commit separado.

MOTIVO:
Readiness compara frescura contra HEAD. Si BATON y WHOAMI se actualizan por separado,
se produce ping-pong de frescura: uno queda fresco y el otro queda viejo.

CONTRATO:
- Actualizar BATON + WHOAMI en el mismo bloque operativo.
- Stagear ambos explícitamente.
- Commit/push.
- Regenerar RADAR.
- Ejecutar readiness final.
- Cerrar solo con READINESS_STATUS OK y Git clean.

MB-GRC-034C_MINIBATTLE_SAFE_LAUNCHER_GATE
==========

DATE..............: 2026-06-21
STATUS............: ACTIVE
LAYER.............: PROJECT_LIFECYCLE_POLICY
PURPOSE...........:
Prevent minibattle execution from damaging terminal state, committing unintended generated outputs,
or bypassing validation/stage discipline when long scripts or launchers are required.

SCOPE.............:
Applies to any minibattle, one-shot, repository patch, session close, session continue, RADAR,
build, validation, or governance patch that requires a long PowerShell script.

SAFE_LAUNCHER_GATE:
Before running a long PowerShell operation, the IA must use the safe launcher pattern:

1. TEMP_SCRIPT_FILE:
   Write long logic to Temp/<MB_ID>.<timestamp>.ps1.

2. SHORT_CONSOLE_LAUNCHER:
   The interactive console only creates the .ps1 file, runs pwsh -File, captures LASTEXITCODE,
   prints SCRIPT_EXIT and LAUNCHER_STATUS, and returns to prompt.

3. NO_INTERACTIVE_TRY_CATCH_FINALLY:
   Do not paste long try/catch/finally blocks directly into the interactive terminal.

4. NO_LAUNCHER_EXIT:
   The launcher must not call exit. It must not close VS Code terminal or the user shell.

5. NO_NESTED_HERE_STRING:
   Do not nest PowerShell here-strings inside launcher here-strings. Use arrays of lines or a
   different construction method.

6. NO_PAGER_CAPTURE:
   Set GIT_PAGER=cat and PAGER=cat. Use git --no-pager for log/diff/show/status diagnostics.

7. BUILD_SCOPE_GATE:
   Do not run BUILD.ps1 unless generated outputs are explicitly in scope. If BUILD dirties
   generated outputs unexpectedly, STOP before commit and decide restore/isolate/commit explicitly.

8. EXPLICIT_STAGE_ONLY:
   Stage only the expected files with explicit git add -- <path>. Do not use git add -A or git add .
   for governance/policy patches.

9. DIRTY_SET_GATE:
   Before commit, compare git status --short against an allowed dirty-file whitelist. If unexpected
   files appear, STOP and report.

10. STAGED_DIFF_GATE:
   Print git --no-pager diff --cached --name-status. If no staged changes exist, STOP before commit.

11. VALIDATION_GATE:
   Run Validate-System.ps1 before commit. If available and relevant, run post-build validation as well.

12. FINAL_CLEAN_GATE:
   After commit/push, verify branch, HEAD, remote head, unpushed/unpulled and clean working tree.

REQUIRED_OUTPUT:
Every lifecycle script must print START/END, LOCAL_TIME, ROOT, PURPOSE, COPY_FROM_HERE, validation
exit code, commit/push exit codes if applicable, FINAL_STATUS, LOG_PATH and NEXT_ACTION.

FAILURE_HANDLING:
- If terminal closes, run recovery diagnostic before continuing.
- If generated outputs become dirty unexpectedly, restore or isolate before further changes.
- If a commit fails because staged set is empty, inspect whether the intended files were actually changed.
- If the shell enters pager END state, rerun with --no-pager and pager environment disabled.

DONE_WHEN.........:
A minibattle can be executed without closing the user's terminal, without accidental generated-output
commits, and with deterministic validation/stage/commit/push/final-clean evidence.
