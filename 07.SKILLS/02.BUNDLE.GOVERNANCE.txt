00.20_CONTEXTO_DEL_PROBLEMA_Y_OBJETIVOS
==========

00.20.01_PROBLEMAS_REALES_QUE_MOTIVAN_RADAR

a) Proyectos crecen y se vuelven inmanejables sin inventario confiable.
b) Tooling e IA fallan cuando el repositorio no tiene mapa.
c) Auditorías, migraciones y refactors requieren trazabilidad.
d) Archivos binarios (xlsx/pdf/png) rompen intentos de "leer todo".
e) Históricos sin disciplina generan caos; sin históricos no hay continuidad.
f) En IA cloud, subir FULL/CORE completo puede disparar costos (COST_INCIDENT).
g) El contenido canónico (HUMAN) y el contenido de skills (SKILLS) necesitan
   compilarse aparte del resto del proyecto para que la IA los entienda como
   unidad completa, sin mezclarlos con código/artefactos operativos.

00.20.02_OBJETIVO_PRINCIPAL

Definir un estándar único para producir vistas:

LITE       : Inventario resumido + diffs básicos (operación diaria).
INDEX      : Inventario total + metadata (para navegación y auditoría).
CORE       : Contenido legible (texto/código) para análisis por IA/humano.
FULL       : Consolidación (INDEX + CORE + extras de auditoría como tree-size).
FULL.HUMAN : Compilación completa del contenido canónico de HUMAN, optimizada
             para lectura por IA.
FULL.SKILLS: Compilación completa del contenido de la carpeta SKILLS, separada
             del resto del proyecto, optimizada para lectura por IA.

00.20.03_OBJETIVO_SECUNDARIO

Asegurar que los outputs puedan:

Ser usados por humanos (lectura rápida, texto plano).

Ser consumidos por IA (chunks y determinismo).

Ser comparables entre ejecuciones (orden estable + formato consistente).

Reducir fricción/costo en cloud evitando "leer todo" sin necesidad.

01.00_DEFINICIONES_BASICAS
==========

01.10_RADAR

RADAR es un estándar de "descripción del FS" (file system) de un proyecto, generado por tooling.

Regla dura:

RADAR describe, no decide.

RADAR no reemplaza decisiones humanas ni lógica de negocio.

----------
01.15_RADAR_LITE
----------

RADAR_LITE es una salida liviana para operación diaria:

- Basada en metadata (similar a INDEX).
- Puede incluir diffs básicos vs ejecución previa (NEW/MODIFIED/DELETED).
- No incluye contenido (no es CORE).
- No interpreta semántica (describe, no decide).
- En primera ejecución (sin histórico previo), no calcula diffs (ver 05.30).

01.20_RADAR_INDEX

Output que lista TODOS los archivos/carpetas relevantes y su metadata (sin contenido).

Incluye, típicamente:

ruta

tamaño

última modificación

tipo lógico

hash (regla por defecto, ver 06.15)

01.30_RADAR_CORE

Output que incluye el contenido SOLO de archivos legibles (texto/código/config).

No incluye:

imágenes, audio, video

pdf (sin OCR)

binarios

xlsx (no se leen celdas)

Nota: la "estructura superficial" de xlsx (nombres de hojas) vive en INDEX (06.30),
no en CORE. CORE nunca incluye contenido de xlsx bajo ninguna forma.

01.40_RADAR_FULL

Output que consolida:

INDEX + CORE

y agrega extras de auditoría (ej.: tree-size por carpeta)

FULL no agrega semántica de negocio; sigue siendo descriptivo.

----------
01.50_HUMAN_Y_SKILLS_COMO_CARPETAS_CANONICAS_(NUEVO)
----------

HUMAN y SKILLS son carpetas canónicas del ecosistema cuya ubicación física
(ruta) NO es definida por RADAR. La ruta de ambas es resuelta vía la skill de
arquitectura de archivos/carpetas del ecosistema (ver DEPENDENCIAS).

RADAR solo:

- Consume la ruta resuelta de HUMAN y SKILLS.
- Compila su contenido en outputs dedicados (FULL.HUMAN, FULL.SKILLS).
- Describe cambios detectados (ver 05.20), sin evaluarlos.

HUMAN es contenido canónico de baja frecuencia de cambio; "de esa carpeta se
desprende todo" y su edición debe evitarse salvo necesidad real. RADAR no
impone ni bloquea esa regla (no decide), solo la hace visible (describe).

02.40_MODO_LOCAL_FS_VS_MODO_AI_UPLOAD
++++++++++

MODO_LOCAL_FS (máquina local / FS completo):

No hay límite de tamaño por archivo fuente para CORE (según versión base).

Se genera output ACTIVE completo.

Si ACTIVE > 8 MB:

se mantiene el ACTIVE intacto,

se generan copias segmentadas <= 8 MB para IA/humano.

MODO_AI_UPLOAD (subida a IA/chat):

Se respeta límite de tamaño por tanda (ej. 8–9MB).

Se suben segmentos (chunks) al chat para consumo por IA.

El activo completo puede no subirse (pero existe localmente).

++++++++++
02.50_POLITICA_RECOMENDADA_DE_USO_(ANTI-COST_INCIDENT)
++++++++++

Regla operativa recomendada:

- Default diario: RADAR_LITE o RADAR_INDEX.
- CORE: solo cuando vas a analizar/editar un módulo específico.
- FULL / FULL.HUMAN / FULL.SKILLS: ocasional (auditoría, "foto completa", o
  análisis canónico puntual), no por costumbre.

Si usas IA cloud:
- Evitar subir FULL/FULL.HUMAN/FULL.SKILLS completos al chat salvo necesidad real.
- Preferir subir segmentos CORE del módulo objetivo.

++++++++++
02.55_ALCANCE_DE_LA_POLITICA_COST_AWARE_(NUEVO)
++++++++++

Aclaración dura para evitar ambigüedad con el contrato de 04.30:

- La política anti-costo (02.50) aplica a QUÉ SE SUBE a IA cloud (chat/upload).
- NO aplica a qué se GENERA en el FS local: los 6 outputs (LITE/INDEX/CORE/FULL/
  FULL.HUMAN/FULL.SKILLS) se generan siempre en cada ejecución, según el
  contrato duro de 04.30.
- Generar localmente es proceso (CPU/disco); subir a IA cloud es costo (tokens).
  RADAR distingue ambos explícitamente.

++++++++++
02.60_NOTA_SOBRE_CONSUMIDORES_(NO_PARTE_DEL_CONTRATO)

RADAR puede ser consumido por sistemas externos (ej: orquestadores IA)

- RADAR sigue describiendo, no decidiendo.
- RADAR no depende de consumidores.

- Usar RADAR_INDEX como default.
- Escalar a CORE por módulo.
- FULL/FULL.HUMAN/FULL.SKILLS solo auditoría o análisis canónico puntual.

06.15_REGLA_DE_HASH_POR_DEFECTO_(NUEVO)
----------

Para evitar ambigüedad y mantener determinismo entre proyectos:

- SHA256 es OBLIGATORIO para todo archivo listado en INDEX cuyo tamaño sea

  NUNCA queda vacío sin explicación: debe registrarse como
  "HASH_OMITIDO_POR_TAMANO".

  metadatos de la ejecución (mismo patrón que la excepción de 04.20).

09.00_TAMANO_SEGMENTACION_Y_CHUNKS
==========

++++++++++
09.10_REGLA_DE_8_MB
++++++++++

Si cualquier output ACTIVE supera 8 MB (incluye FULL.HUMAN y FULL.SKILLS):

El archivo ACTIVE se mantiene intacto.

Se crean copias segmentadas <= 8 MB.

++++++++++
09.20_NAMING_DE_SEGMENTOS
++++++++++

Determinístico.

Sin la palabra "part".

Ejemplo:
RADAR_FULL.ACTIVE.txt.seg.001
RADAR_FULL.ACTIVE.txt.seg.002
...

----------
09.25_NAMING_DE_HISTORICOS_(NUEVO)
----------

Determinístico, basado en timestamp ISO sin separadores ambiguos (formato
AAAAMMDD-HHMMSS).

El ACTIVE de cada output siempre se sobrescribe en la ejecución más reciente.
El HIST nunca se sobrescribe: al generarse un nuevo ACTIVE, el ACTIVE anterior
se mueve a histórico con el siguiente naming:

RADAR_<TIPO>.HIST.<timestamp>.txt

Ejemplo:
RADAR_FULL.HIST.20260226-143000.txt
RADAR_FULL.HUMAN.HIST.20260226-143000.txt
RADAR_FULL.SKILLS.HIST.20260226-143000.txt

Los 6 outputs de una misma corrida comparten el mismo timestamp, permitiendo
agruparlos por prefijo como pertenecientes a una única ejecución.

++++++++++
09.30_SEGMENTACION_HIBRIDA
++++++++++

Texto (outputs y/o contenido textual):

Preferir corte por límites de línea (mejor lectura humana).

Si no es posible (líneas extremadamente largas), fallback a corte por bytes.

Binario (si se segmenta por alguna razón excepcional):

Corte por bytes (reconstrucción byte-exact).

Notas:

El objetivo de la segmentación es consumo IA/humano, no optimización.

El activo completo es la fuente de verdad local.

La actualización de backlog debe apuntar al backlog operativo del proyecto activo, si existe.

Si existen varios backlogs, la IA debe usar el backlog operativo principal.
Si no puede identificarlo con evidencia, debe declararlo como brecha y pedir instrucción.

----------
06.60_DEUDA_TECNICA_EN_CIERRE
----------

La deuda técnica debe registrarse como backlog operativo, salvo que el proyecto tenga un artefacto formal de deuda técnica.

Debe distinguirse:

- deuda técnica;
- backlog funcional;
- brecha de información;
- riesgo abierto.

----------
06.70_HUMAN_EN_CIERRE
----------

HUMAN solo se actualiza cuando existe:

- cambio doctrinal;
- cambio de precedencia;
- cambio de intención humana;
- cambio de regla raíz;
- instrucción explícita del humano.

HUMAN no se actualiza por cambios operativos normales.

----------
06.80_PATH_MINIBATTLES_BATTLES_ENTREGABLE
----------

El cierre debe incluir path accionable de desarrollo:

MINIBATTLE -> BATTLE -> ENTREGABLE_FUNCIONAL_A_GERENCIA

Formato recomendado:

MINIBATTLE | OBJETIVO | OUTPUT | BATTLE | ENTREGABLE_GERENCIA | ESTADO

Si faltan fechas, responsables o criterios de éxito, no se inventan.
Se declaran como pendientes solo si son necesarios para gestión.

----------
06.90_EVIDENCIA_Y_NO_INVENCION
----------

No se debe inventar estado.

Si falta evidencia, la IA debe declararlo como brecha.

Hechos confirmados, inferencias y pendientes deben distinguirse.

==========
07.00_ACTUALIZACION_DE_ARTEFACTOS
==========

----------
07.10_REGLA_BASE
----------

Las actualizaciones de continuidad se hacen al cierre formal o bajo instrucción explícita.

Artefactos típicos:

- BATON principal del proyecto activo;
- backlog operativo del proyecto activo;
- deuda técnica;
- HUMAN, solo si aplica;
- WhoAmI, solo si cambia contexto humano operacional;
- RADAR, si aplica;
- readiness o equivalente de cierre, si existe.

----------
07.20_NO_ACTUALIZACION_PERMANENTE
----------

Antes y durante la sesión no se obliga a actualizar:

- BATON;
- backlogs;
- deuda técnica;
- HUMAN.

Esto no prohíbe actualizar estos artefactos si el humano lo solicita explícitamente.

----------
07.30_CRITERIO_DE_ACTUALIZACION
----------

Actualizar solo si:

- hubo cambio real;
- hay evidencia suficiente;
- el artefacto existe o el humano aprueba crearlo;
- el cambio no duplica información innecesaria.

==========
08.00_VALIDACION_CON_RADAR
==========

----------
08.10_REGLA_GENERAL
----------

RADAR es mapa/evidencia de filesystem, no fuente de verdad lógica.

RADAR se usa para:

- validar rutas;
- detectar archivos creados, modificados o eliminados;
- revisar estado documental;
- apoyar cierre de sesión;
- preparar continuidad para IA en blanco.

----------
08.20_CUANDO_PEDIR_RADAR
----------

Si falta RADAR vigente o evidencia suficiente para cerrar con fidelidad, la IA debe pedir ejecutar RADAR antes de completar el cierre formal.

Esto aplica especialmente cuando:

- hubo cambios de filesystem;
- se crearon archivos;
- se modificaron artefactos;
- se eliminó o movió contenido;
- el cierre requiere confirmar rutas;
- el proyecto depende de RADAR como mapa de continuidad.

----------
08.30_CUANDO_RADAR_PUEDE_NO_APLICAR
----------

RADAR puede no aplicar si:

- la sesión fue puramente conceptual;
- no hubo cambios de filesystem;
- el proyecto no usa RADAR ni equivalente;
- el humano pide cierre parcial y acepta la brecha.

En ese caso se debe declarar explícitamente que RADAR no fue usado.

==========
09.00_RELACION_CON_GIT
==========

Este Skill no define comandos Git.

Toda operación Git debe derivarse al Skill de Git aplicable del proyecto activo.

Ejemplos de materias derivadas al Skill de Git:

- git status;
- git diff;
- git add;
- git commit;
- git push;
- ramas;
- tags;
- reversión;
- staged vs unstaged;
- archivos ignorados;
- commits atómicos.

Durante cierre, este Skill puede exigir verificar estado de versionamiento,
pero no redefine cómo hacerlo.

==========
10.00_REGLAS_DE_CONSISTENCIA
==========

Caso 1:
BATON contradice HUMAN.

Dictamen:
BATON está incorrecto o desactualizado.

Acción:
Actualizar BATON o declarar brecha.

----------

Caso 2:
Skill contradice HUMAN.

Dictamen:
Conflicto conceptual.

Acción:
Resolver actualizando explícitamente HUMAN o Skill según decisión humana.

----------

Caso 3:
RADAR contradice BATON.

Dictamen:
BATON está desactualizado o RADAR refleja cambios no incorporados.

Acción:
Revisar evidencia antes de cerrar.

----------

Caso 4:
IA propone acción que viola Skill.

Dictamen:
Acción inválida.

Acción:
Bloquear o pedir excepción humana documentada.

----------

Caso 5:
Usuario pide estado pero no cierre.

Dictamen:
No activar cierre formal.

Acción:
Responder estado, pero no ejecutar protocolo completo.

----------

Caso 6:
Usuario dice "cierra sesión".

Dictamen:
Activar cierre formal completo.

Acción:
Ejecutar contrato de cierre de este Skill.

==========
11.00_ANTIPATRONES
==========

- Actualizar BATON en cada microavance sin necesidad.
- Convertir el cierre en burocracia permanente.
- Usar RADAR como fuente de verdad lógica.
- Cerrar sesión sin declarar brechas.
- Inventar estado para completar cierre.
- Actualizar HUMAN por cambios operativos normales.
- Mezclar Git dentro de este Skill.
- Duplicar información entre BATON, backlog y HUMAN.
- Crear backlog si el proyecto ya tiene uno operativo suficiente.
- Omitir path minibattles -> battles -> entregable funcional a gerencia al cierre formal.
- Tratar "estado del proyecto" como "cierra sesión".

==========
12.00_DEFINICIONES_OPERATIVAS
==========

----------
12.10_MINIBATTLE
----------

Unidad corta de avance, verificable, con output concreto y evidencia mínima.

Debe reducir riesgo, desbloquear trabajo o producir avance funcional.

----------
12.20_BATTLE
----------

Bloque mayor de trabajo compuesto por una o más minibattles.

Debe conducir hacia una capacidad funcional, decisión relevante o entregable demostrable.

----------
12.30_ENTREGABLE_FUNCIONAL_A_GERENCIA
----------

Resultado demostrable, entendible por gerencia y con valor visible.

Puede ser demo, reporte, tablero, presentación, proceso operativo validado o automatización funcional.

No debe ser solo documentación interna salvo que el objetivo gerencial sea gobierno documental.

----------
12.40_BATON_PRINCIPAL_DEL_PROYECTO_ACTIVO
----------

Archivo o artefacto de continuidad que resume el estado operativo mínimo para continuar el proyecto en una IA en blanco.

----------
12.50_BACKLOG_OPERATIVO_DEL_PROYECTO
----------

Lista activa de pendientes, mejoras, deuda técnica, riesgos y próximas acciones del proyecto.

Se actualiza al cierre si existe y si hay información suficiente.

----------
12.60_RADAR
----------

Mapa o índice vigente del filesystem/proyecto usado como evidencia de estado, rutas y artefactos disponibles.

Si falta RADAR y el cierre depende de estado de archivos, la IA debe pedir su ejecución.

==========
13.00_ACUMULACION_Y_PERFECCIONAMIENTO_DE_ARTEFACTOS
==========

----------
13.10_REGLA_BASE
----------

Durante el cierre formal de sesión, cada artefacto debe cumplir su función correcta.

No se debe copiar la misma información en todas las capas.

Regla de capa:

- Backlog acumula.
- Deuda técnica acumula.
- HUMAN se perfecciona.
- BATON se refresca.
- RADAR se regenera.
- Git registra.

----------
13.20_BACKLOG
----------

El backlog operativo del proyecto acumula pendientes no resueltos, mejoras, decisiones abiertas y próximos pasos.

En cada cierre formal:

- se agregan nuevos pendientes;
- se actualizan pendientes existentes;
- se cierran ítems resueltos;
- se fusionan duplicados;
- se reclasifican ítems cuando cambia su naturaleza.

Un pendiente no resuelto permanece abierto.

No debe duplicarse como nuevo ítem si ya existe.

----------
13.30_DEUDA_TECNICA
----------

La deuda técnica acumula problemas técnicos, estructurales u operativos no resueltos.

Debe distinguirse de:

- backlog funcional;
- brecha de información;
- riesgo abierto;
- decisión pendiente.

Estados recomendados:

- NUEVA
- EXISTENTE_SIN_CAMBIO
- EXISTENTE_ACTUALIZADA
- MITIGADA
- RESUELTA
- ACEPTADA_CONSCIENTEMENTE
- DESCARTADA

La deuda técnica debe mantener, cuando exista evidencia suficiente:

- descripción;
- causa;
- impacto;
- riesgo;
- evidencia;
- próximo paso;
- prioridad;
- relación con minibattle o battle si aplica.

----------
13.40_HUMAN
----------

HUMAN no acumula tareas ni pendientes operativos.

HUMAN se perfecciona solo cuando existe:

- cambio doctrinal;
- aclaración de intención humana;
- cambio de visión;
- cambio de principio rector;
- cambio de precedencia;
- cambio de regla raíz;
- mejora conceptual canónica.

Ejemplo correcto:
Actualizar HUMAN para declarar una nueva regla superior de arquitectura o gobernanza.

Ejemplo incorrecto:
Agregar a HUMAN una tarea pendiente de implementación.

Las tareas pendientes van en backlog, no en HUMAN.

----------
13.50_BATON
----------

BATON no acumula historia completa.

BATON es snapshot operativo vigente.

Debe reflejar:

- estado actual;
- objetivo activo;
- próximos pasos;
- riesgos o brechas que afectan continuidad;
- archivos/capas relevantes para continuar;
- decisión operativa vigente.

BATON debe refrescarse al cierre formal cuando exista información suficiente.

No debe convertirse en manual, backlog ni changelog completo.

----------
13.60_RADAR
----------

RADAR no se edita manualmente.

RADAR se regenera como evidencia del filesystem o estructura equivalente del proyecto.

RADAR no decide lógica ni doctrina.

RADAR ayuda a validar:

- archivos existentes;
- rutas;
- cambios de filesystem;
- outputs;
- estado documental observable.

----------
13.70_GIT
----------

Git registra trazabilidad histórica mediante commits, ramas, tags y diferencias.

Este Skill no define comandos Git.

La operación Git debe derivarse al Skill de Git aplicable.

----------
13.80_REGLA_ANTI_DUPLICACION
----------

No se debe duplicar la misma información en múltiples artefactos.

Cada contenido debe vivir en su capa correcta:

- doctrina en HUMAN;
- estado operativo vigente en BATON;
- pendientes en backlog;
- problemas estructurales en deuda técnica;
- evidencia filesystem en RADAR;
- trazabilidad histórica en Git.

Si una misma idea aparece en varias capas, debe tener propósito distinto en cada una.

----------
13.90_CRITERIO_DE_CIERRE
----------

Al cierre formal, la IA debe revisar si cada dato pertenece a:

- HUMAN;
- BATON;
- backlog;
- deuda técnica;
- RADAR;
- Git;
- otro artefacto del proyecto activo.

Si no existe evidencia suficiente para clasificar un dato, debe declararse como brecha y no inventarse.

00.20_CONTEXTO_Y_OBJETIVO
==========

Este skill existe porque muchos prompts fallan antes de escribirse.

El error común no es redactar mal, sino:
- no entender el problema;
- no definir alcance;
- no identificar inputs;
- no definir outputs;
- no declarar criterios de éxito;
- no declarar criterios de rechazo;
- automatizar antes de comprender;
- pedir ejecución sin evidencia;
- convertir supuestos en hechos.

Objetivo:
Forzar una fase previa de discovery, challenge y readiness antes de generar prompts de impacto.

==========

Aplicar este skill cuando el usuario solicite:

- crear un prompt relevante;
- mejorar un prompt existente;
- auditar un prompt existente;
- crear un prompt para otra IA;
- crear un prompt reutilizable;
- crear prompt para repositorio, documento, planilla, presentación, sistema, código o workflow;
- crear o perfeccionar un skill, BATON, RADAR, runbook, workflow o prompt de ejecución técnica;
- revisar output de otra IA;
- preparar una IA en blanco para operar con contexto controlado.

No aplicar con máxima fricción cuando:
- la tarea sea simple;
- el usuario pida una traducción directa;

05.00_PROMPT_READINESS_GATE
==========

Antes de generar prompt final, la IA debe declarar uno de estos estados:

PROMPT_READY = NO_BLOCKED
Existen dudas bloqueantes. No generar prompt final.

PROMPT_READY = NO_CAN_ITERATE

00.00 PURPOSE
==========

DragonflyFocus is a mandatory focus-control GRC for SkillsMachine operations.

Its purpose is to prevent conversational drift, uncontrolled branching, overproduction,
premature tooling, and assistant-led deviation from the HUMAN-defined objective.

DragonflyFocus exists because the user may produce many valuable ideas during project work,
but not every idea is allowed to become the active workstream.

The assistant must capture side ideas, classify them, and return to the active objective.

==========
01.00 CORE RULE
==========

HUMAN defines the objective.

The assistant must not replace the active objective because of conversational drift,
interesting side ideas, technical curiosity, tooling opportunities, or assistant preference.

If the user introduces a new idea, the assistant must classify it before acting.

No change of mission is allowed without explicit human confirmation.

==========
02.00 MANDATORY FOCUS_GATE
==========

Every operational response under the covered usecases must include or internally apply:

FOCUS_GATE:
- OBJECTIVE_ACTIVE:
- CURRENT_STEP:
- USER_NEW_IDEA:
- CLASSIFICATION:
- DECISION:
- NEXT_ACTION:

CLASSIFICATION values:

- CORE_OBJECTIVE
  The new idea directly advances the active objective.

- BLOCKER
  The new idea reveals something that blocks the active objective.

- TECHNICAL_DEBT
  The new idea is valid but should be registered and not executed now.

- BACKLOG
  The new idea is useful but not needed for the active objective.

- CHANGE_OF_MISSION
  The new idea would replace or materially alter the active objective.

- DISCARD
  The new idea is noise, duplicate, obsolete, or not worth carrying forward.

==========
03.00 DECISION RULES
==========

03.10 CORE_OBJECTIVE

If classification is CORE_OBJECTIVE:
- incorporate it into the current step;
- explain why it advances the objective;
- keep the next action singular.

03.20 BLOCKER

If classification is BLOCKER:
- stop execution;
- define the blocker;
- define the minimum unblock action;
- do not continue with unrelated tasks.

03.30 TECHNICAL_DEBT

If classification is TECHNICAL_DEBT:
- register it as technical debt;
- do not execute it immediately;
- return to OBJECTIVE_ACTIVE.

03.40 BACKLOG

If classification is BACKLOG:
- capture it briefly;
- do not expand it;
- return to OBJECTIVE_ACTIVE.

03.50 CHANGE_OF_MISSION

If classification is CHANGE_OF_MISSION:
- stop;
- explicitly ask the human to confirm mission replacement;
- do not proceed until confirmation is given.

Required wording:

CAMBIO_DE_MISION detectado.
Objetivo activo actual:
[OBJECTIVE_ACTIVE]

Nuevo objetivo propuesto:
[NEW_OBJECTIVE]

Impacto:
[IMPACT]

Confirmas reemplazar el objetivo activo? YES/NO

03.60 DISCARD

If classification is DISCARD:
- state briefly why it is discarded;
- do not preserve it unless the human insists.
